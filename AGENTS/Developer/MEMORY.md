### Technical Decisions & Rationale

**1. Color Handling**: HTML-style tags (`<red>text</red>`)
- Rationale: Markdown has no native color support
- Benefit: Lossless round-trip conversion
- Trade-off: Not pure markdown, but opt-out available

**2. Page Links**: Preserve Notion IDs (`[[page_id]]`)
- Rationale: Enables future reverse resolution to titles/URLs
- Benefit: Maintains reference integrity
- Alignment: Matches existing markdown-parser format

**3. Table Format**: GitHub-flavored markdown
- Rationale: Widely supported, human-readable
- Benefit: Compatible with existing parser
- Implementation: Pipe-delimited with header separator

**4. Error Handling**: Graceful degradation
- Rationale: Don't fail entire conversion on single unsupported block
- Benefit: Robust conversion even with edge cases
- Implementation: Placeholder comments for unsupported types

**5. API Design**: Class + helper pattern
- Rationale: Flexibility for complex use cases, convenience for simple ones
- Benefit: Works for both one-off conversions and batch processing
- Example: `NotionToMarkdownConverter` class + `notionToMarkdown()` helper
