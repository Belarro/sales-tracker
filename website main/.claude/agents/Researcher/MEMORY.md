## Claude Agent SDK Memory Research - 2025-10-08

### Research Mission Completed
**Objective**: Investigate agent memory management patterns in Claude Agent SDK
**Scope**: 14 files, 2,823 lines in ~/.claude/examples/sdk/
**Status**: 100% complete, all files read fully with evidence
**Session Report**: Sessions/CollaborativeIntelligence-2025-10-08.md

### Key Findings

#### 1. Memory Architecture: Conversation-Based, Not Store-Based
- **No explicit memory API** - Memory = conversation context
- **Sessions** (08-sessions.ts): Primary persistence via session_id
- **System prompts** (01-core-patterns.ts): Agent knowledge injection
- **Stateful tools** (02-custom-tools.ts): Custom in-memory stores

#### 2. Context Injection Patterns Identified
1. **System Prompt Injection** (lines 421-439, 01-core-patterns.ts)
2. **Context String Embedding** (lines 252-262, 04-subagent-essentials.ts)
3. **Agent Definition Prompts** (lines 81-109, 04-subagent-essentials.ts)
4. **Multi-Turn Streaming** (lines 84-118, 01-core-patterns.ts)
5. **Session Persistence** (lines 9-60, 08-sessions.ts)
6. **Stateful Tools** (lines 161-224, 02-custom-tools.ts)

#### 3. Memory Limits and Cost Management
- **Turn limits** (maxTurns): 1 (simple) → 5-10 (tools) → 20-30 (complex)
- **Cost tracking** (07-cost-tracking.ts): Budget-based memory constraints
- **Model degradation** (lines 115-158, 07-cost-tracking.ts): Opus → Sonnet → Haiku

#### 4. State Management Between Turns
- **parent_tool_use_id tracking** (04-subagent-essentials.ts, lines 112-132)
- **Turn counting** (01-core-patterns.ts, lines 520-542)
- **Conversation accumulation** - State builds through message history

#### 5. Anti-Patterns Documented
1. String prompts for multi-turn (limits functionality)
2. No maxTurns limit (unbounded costs)
3. Ignoring cost tracking (no visibility)
4. bypassPermissions in production (security risk)

### Implementation Recommendations

**For CollaborativeIntelligence Agents**:
1. Use session IDs: `agent-{name}-{timestamp}`
2. Inject context via system prompts (custom or preset with append)
3. Pass context explicitly to subagents (don't assume shared memory)
4. Track costs via modelUsage and total_cost_usd
5. Set maxTurns based on complexity (5-30 range)
6. Use AsyncIterable for dynamic context
7. Implement stateful tools for persistent data

#### Three Primary Approaches Researched

**1. Hierarchical Working Memory Management (HiAgent)**
- **Source**: ACL 2025, arXiv:2408.09559 (August 2024)
- **Innovation**: Subgoal-based memory chunking within working memory
- **Architecture**: Cross-trial memory (long-term) + In-trial hierarchical memory (subgoals)
- **Performance**: 200% success rate improvement, 3.8x fewer steps required
- **Relevance**: Demonstrates hierarchical organization within working memory tier

**2. Modular Procedural Memory Allocation (LEGOMem)**
- **Source**: arXiv:2510.04851 (October 2024)
- **Innovation**: Role-based memory tier separation across multi-agent systems
- **Architecture**: Full-task memories (orchestrator) + Subtask memories (agent level)
- **Performance**: Enables smaller LLMs to narrow performance gaps
- **Relevance**: Validates separation of strategic vs. tactical memory

**3. Dynamic Multi-Tier Production System (Mem0)**
- **Source**: arXiv:2504.19413 (April 2025) + Industry implementation (mem0.ai)
- **Innovation**: OS-inspired memory paging with graph-based enhancement
- **Architecture**: Context window + Vector-based long-term + Graph-based relational
- **Performance**: 26% accuracy improvement, 91% lower latency, 90%+ cost savings
- **Relevance**: Production-validated multi-tier architecture with real-world deployment

1. **Hierarchical Organization**: Strategic/planning memory separated from tactical/execution
2. **Selective Retrieval**: Minimize context window usage through targeted loading
3. **Dynamic Management**: Proactive replacement of obsolete memory
4. **Separation of Concerns**: Different memory types for different functions
5. **Context Window Optimization**: Reduce redundancy within fixed constraints
6. **Autonomous Operation**: LLM agents manage their own memory via function calls
