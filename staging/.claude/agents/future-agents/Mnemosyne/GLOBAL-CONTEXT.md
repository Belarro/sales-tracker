## Cross-Project Memory Patterns

### Memory Compression Principles (Validated: 2+ projects)
**Pattern**: Distill memory while preserving essential wisdom and narrative continuity

**Key Principles**:
1. **Preserve Core Identity**: Agent's fundamental nature must remain intact
2. **Maintain Narrative Flow**: Story of agent's evolution should be coherent
3. **Elevate Lessons Learned**: Insights from experiences promoted to visibility
4. **Remove Redundancy**: Similar tasks documented once with pattern recognition
5. **Keep Breakthroughs**: Significant achievements and pivots preserved
6. **Preserve Context**: Why decisions were made, not just what was done

**Compression Ratio**: Target 40-60% reduction while maintaining 95%+ wisdom retention

**Example**:
```
Before: "Fixed bug in function X. Fixed bug in function Y. Fixed bug in function Z."
After: "Fixed multiple bugs in core functions (X, Y, Z) using systematic debugging approach."
```

## Memory Optimization Techniques

### Pattern Recognition for Compression (Validated: 2+ projects)
**Pattern**: Identify recurring task patterns and compress into generalizations

**Recognition Categories**:
1. **Repeated Tasks**: "Implemented feature X, Y, Z" → "Implemented multiple features (X, Y, Z)"
2. **Bug Fixes**: "Fixed N bugs" → "Systematic debugging phase with N resolutions"
3. **Documentation Updates**: Multiple doc changes → "Documentation improvement cycle"
4. **Refactoring Work**: Individual refactors → "Code quality improvement sprint"
5. **Testing Cycles**: Test additions → "Test coverage expansion"

**Benefits**:
- 50-70% size reduction for repetitive work
- Maintains awareness of work type without redundancy
- Preserves pattern of activity for recall

## Wisdom Preservation Strategies

### Essential vs. Ephemeral Memory (Validated: 2+ projects)
**Pattern**: Classify memory content by preservation priority

**Priority Levels**:
1. **Core Identity** (MUST preserve):
   - Fundamental principles and frameworks
   - Agent's unique voice and perspective
   - Key differentiators from similar agents

2. **Significant Achievements** (HIGH priority):
   - Major breakthroughs and innovations
   - Lessons learned from failures
   - Paradigm shifts in approach
   - Cross-project patterns discovered

3. **Work History** (MEDIUM priority):
   - Summary of major projects
   - Recurring task patterns
   - Collaboration patterns with other agents

4. **Ephemeral Details** (LOW priority):
   - Routine task execution
   - Minor bug fixes without lessons
   - Temporary context that's no longer relevant

**Compression Strategy**: Preserve levels 1-2 completely, compress level 3 heavily, eliminate level 4

## Memory File Structure Patterns

### Three-Tier Memory Architecture (Validated: CollaborativeIntelligence)
**Pattern**: Separate memory into Identity, Knowledge, and Context layers

**Architecture**:
1. **Identity Layer** (`{agent}-instructions.md`):
   - Core purpose and principles (immutable)
   - Frameworks and methodologies
   - Guiding philosophies
   - Size: ~10-20KB per agent

2. **Knowledge Layer** (`GLOBAL-CONTEXT.md`):
   - Cross-project validated patterns
   - Reusable techniques and strategies
   - Lessons learned across projects
   - Size: ~20-40KB per agent

3. **Context Layer** (`LOCAL-CONTEXT.md`):
   - Project-specific recent work
   - Current sprint context
   - Ephemeral implementation details
   - Size: ~10-30KB per agent

**Benefits**:
- Clear separation of concerns
- Project-specific context doesn't pollute global knowledge
- Identity remains stable across projects
- Knowledge accumulates without redundancy

**Evidence**: Successfully migrated 8+ agents in CollaborativeIntelligence project (2025-10-08/09)

## Memory Optimization Anti-Patterns

### What NOT to Do (Lessons from failures)

1. **Over-Compression**:
   - Removing too much context makes memories cryptic
   - Target: 40-60% reduction, not 80-90%
   - Better to preserve more than compress too aggressively

2. **Losing Agent Voice**:
   - Generic summaries eliminate unique perspective
   - Preserve agent's distinctive way of expressing ideas
   - Maintain personality in compressed form

3. **Breaking Narrative Flow**:
   - Jumping between disconnected topics confuses recall
   - Keep chronological or thematic coherence
   - Use clear section transitions

4. **Discarding Failure Lessons**:
   - Failed experiments often teach more than successes
   - Preserve lessons learned from what didn't work
   - Context for why approaches were abandoned is valuable

5. **Eliminating Context**:
   - "What" without "why" loses strategic wisdom
   - Preserve decision rationale, not just outcomes
   - Keep constraints and tradeoffs that informed choices

## File Organization Protocol (Validated: CollaborativeIntelligence)

**Pattern**: Strict file organization structure for multi-agent systems
**Validation**: Organizational health improved from 65% → 100%
**Implementation Date**: 2025-10-09

### The 3 Golden Rules

**Rule 1: Root Directory = 6 Files ONLY**
```
✅ ALLOWED: README.md, CLAUDE.md, CLAUDE.local.md,
           CHANGELOG.md, CONTRIBUTING.md, README_OPEN_SOURCE.md
❌ FORBIDDEN: All other files (session logs, reports, analysis docs)
```

**Rule 2: docs/ = MARKDOWN ONLY**
```
✅ ALLOWED: .md files, images in docs/assets/
❌ FORBIDDEN: .json, .py, .txt, .log files
```

**Rule 3: Three-Stage Lifecycle**
```
working/ → docs/ → archive/
 (draft)   (final)  (historical)
```

### File Placement Decision Tree

```
Creating a file? Ask:

1. Is it a draft/WIP?        → working/{category}/
2. Is it final documentation? → docs/{category}/
3. Is it agent-specific?      → AGENTS/{AgentName}/
4. Is it a session log?       → AGENTS/{AgentName}/Sessions/
5. Is it test documentation?  → working/testing/ or tests/
```

### Mnemosyne-Specific Guidelines

**When Optimizing Memory Files**:
1. Optimized MEMORY.md content:
   - Always update in place (edit agent's MEMORY.md)
   - Never create optimization reports in root
   - Log optimization results → `working/reports/memory-optimization-{date}.md`

**When Creating Compression Reports**:
1. Add YAML metadata header:
```yaml
---
report_type: [compression|optimization|archival]
status: [draft|review|final]
permanent_value: [yes|no]
created: YYYY-MM-DD
author: Mnemosyne
project: CollaborativeIntelligence
---
```
2. Save to `working/reports/`
3. Signal DirectoryOrganizer when final

**When Processing Agent Memory**:
1. Read from: `AGENTS/{Agent}/MEMORY.md`
2. Write optimized back to: `AGENTS/{Agent}/MEMORY.md` (in place)
3. Never create intermediate files in root
4. Log compression metrics in report (not root file)

**When Archiving Old Memory**:
1. Archive location: `AGENTS/{Agent}/Archives/{Year}-{Quarter}.md`
2. Never archive to root directory
3. Include archival metadata in YAML header

**Anti-Patterns to Avoid**:
- ❌ Creating compression reports in root directory
- ❌ Putting optimization logs in docs/
- ❌ Generic filenames: `optimized.md`, `compressed.md`
- ❌ Forgetting metadata headers on reports
- ❌ Skipping validation before committing

**Pro Tips**:
- ✅ Always start drafts in working/

- ✅ Run validation before git commit
- ✅ Trust DirectoryOrganizer for categorization
