## Domain-Specific Patterns

### Application Analysis Patterns
1. **Architecture Mapping**
   - Start with high-level component overview
   - Identify core user flows and interactions
   - Map data relationships and state management
   - Document technical constraints early
   - Understand performance implications

2. **User Journey Mapping**
   - Define primary user personas
   - Map critical user paths
   - Identify decision points and pain areas
   - Document interaction patterns
   - Analyze task completion flows

3. **Design Opportunity Identification**
   - Audit existing interface inconsistencies
   - Find areas of user friction
   - Identify accessibility gaps
   - Spot performance optimization opportunities
   - Locate areas for visual enhancement

### Visual Design Patterns
1. **Minimalist Design Principles**
   - Use white space intentionally
   - Limit color palette to essentials
   - Choose typography deliberately
   - Remove decorative elements that don't serve function
   - Focus on content hierarchy

2. **Component Design Patterns**
   - Start with atomic elements
   - Build up to molecules and organisms
   - Ensure component flexibility
   - Design for multiple contexts
   - Document usage guidelines

3. **Responsive Design Strategies**
   - Design mobile-first when appropriate
   - Use fluid typography scales
   - Implement flexible grid systems
   - Consider touch targets for mobile
   - Test across device categories

### Design System Patterns
1. **Token Architecture**
   - Define primitive values first
   - Create semantic aliases
   - Ensure token scalability
   - Document token relationships
   - Version token changes

2. **Component Documentation**
   - Include visual examples
   - Provide code snippets
   - Document props and variants
   - Show do's and don'ts
   - Include accessibility notes

## Best Practices

### Application Understanding
1. Spend time studying the codebase architecture
2. Interview developers about technical constraints
3. Observe real users interacting with the application
4. Document existing patterns before proposing changes
5. Understand performance implications of design decisions

### Design Process
1. Always start with low-fidelity explorations
2. Test designs with real data, not lorem ipsum
3. Consider edge cases and error states
4. Design for accessibility from the start
5. Iterate based on user feedback

### Visual Design
1. Maintain consistent spacing throughout
2. Use color purposefully, not decoratively
3. Ensure sufficient contrast ratios
4. Design for clarity over cleverness
5. Test designs at multiple scales

### Design Systems
1. Start small and expand gradually
2. Document as you build
3. Version control design assets
4. Create living style guides
5. Enable easy developer adoption

### Collaboration
1. Communicate design decisions clearly
2. Provide detailed specifications
3. Be open to technical constraints
4. Participate in code reviews
5. Maintain design-development alignment

- **Initial Observation**: Designs created without deep app understanding often miss the mark
- **Learning**: Invest significant time in application analysis upfront
- **Implementation**: Created structured discovery phase in workflow

### Simplicity vs. Functionality
- **Initial Observation**: Overly minimal designs sometimes sacrificed usability
- **Learning**: Balance aesthetics with clear functionality
- **Implementation**: Developed "functional beauty" principle

### Component Flexibility
